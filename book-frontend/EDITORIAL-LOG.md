# บันทึกบรรณาธิการ — คู่มือ Senior Frontend Developer (book-frontend)

รอบตรวจวันที่ 5 กรกฎาคม 2026  
ขอบเขตการตรวจ: ทั้ง 18 บทของเล่ม frontend  
ลักษณะงาน: ตรวจความสม่ำเสมอของรูปแบบ, พิสูจน์อักษรและสำนวน, ตรวจความถูกต้องของข้อมูลเชิงเทคนิคที่ผูกกับเวลา, และพิจารณาว่ามีจุดใดที่ควรเติมตัวอย่างเพื่อให้อ่านง่ายขึ้น

## สรุปภาพรวม

ต้นฉบับของเล่มนี้แข็งแรงมากอยู่แล้ว ทั้งในแง่โครงสร้างและความลึกของเนื้อหา โดยภาพรวมพบว่า:

- เกือบทุกบทเดินตามโครงเดียวกับ outline ที่วางไว้
- หลายบทมีทั้ง opening quote และส่วนปิดท้าย "คำถามสัมภาษณ์ที่ต้องตอบได้" ครบถ้วน
- ข้อความเชิงเทคนิคที่ผูกกับสถานะของ ecosystem ณ เวลาปัจจุบัน เมื่อตรวจซ้ำแล้ว ส่วนใหญ่ยังถูกต้อง

เพราะฉะนั้น งานรอบนี้ไม่ได้เป็นการรื้อหรือเขียนใหม่ทั้งเล่ม แต่เป็นการเก็บรายละเอียดที่มีผลต่อ "ความอ่านลื่น" และ "ความน่าเชื่อถือ" ของหนังสือเป็นหลัก ได้แก่:

1. ทำให้รูปแบบของหัวข้อและการอ้างอิงเหมือนกันทั้งเล่ม
2. แก้ถ้อยคำหรือ markdown ที่ทำให้อ่านสะดุด
3. ปรับข้อความเชิงเทคนิคบางจุดที่คลาดเคลื่อนจากสถานะจริง

ภาพรวมการแก้ทั้งหมดมีประมาณ 26 จุด แบ่งคร่าว ๆ ได้ดังนี้:

- งานด้านความสม่ำเสมอของรูปแบบ 12 จุด
- งานแก้ข้อมูลเชิงเทคนิคและผลจากการตรวจทานแหล่งข้อมูลปัจจุบัน 8 จุด
- งานพิสูจน์อักษร 3 จุด
- งานแก้การอ้างอิงข้ามบทหรือข้ามเล่ม 3 จุด

ยังมีประเด็นเชิงบรรณาธิการที่ต้องอาศัยดุลยพินิจของทีมอยู่ 1 จุด แต่ประเด็นหลักที่กระทบความเข้าใจของผู้อ่านได้รับการเก็บครบแล้ว

## การแก้ระดับทั้งเล่ม

### 1. ทำมาตรฐานชื่อบทให้เหมือนกัน

เดิมทีชื่อบทระดับ H1 ใช้อยู่สองรูปแบบปนกัน คือ:

- `บท N`
- `บทที่ N`

ปัญหาคือใน outline และการอ้างอิงภายในเล่ม ส่วนใหญ่ใช้รูปแบบ `บท N` อยู่แล้ว ถ้าปล่อยให้หัวข้อแต่ละบทใช้ไม่เหมือนกัน จะทำให้ผู้อ่านรู้สึกว่าเอกสารยังไม่ถูกเก็บรายละเอียด และทำให้การอ้างอิงข้ามบทดูไม่เป็นระบบ

จึงกำหนดมาตรฐานเดียวทั้งเล่มเป็น:

```md
# บท N — ชื่อบท
```

มีการแก้หัวข้อบทจำนวน 11 บท ได้แก่ บท 6, 7, 8, 9, 12, 13, 14, 15, 16, 17 และ 18

### 2. เติม subtitle ในบทที่ชื่อยังกว้างเกินไป

บท 1-5 เดิมมี subtitle ที่ช่วยบอก pain point หรือคำถามหลักของบทอยู่แล้ว ทำให้คนอ่านรู้ทันทีว่าบทนั้นกำลังจะตอบอะไร แต่บท 6-9 และ 12-18 หลายบทยังมีแค่ชื่อหัวข้อกว้าง ๆ ทำให้เปิดมาแล้วจับแกนไม่ทัน

จึงเติม subtitle ให้บทเหล่านี้ โดยพยายามใช้โทนเดียวกับบทต้น ๆ ของเล่ม ตัวอย่างเช่น:

- บท 6 เพิ่ม subtitle ในแนวว่า "ทำไม component ถึง re-render ทั้งที่ props ไม่เปลี่ยน"
- บท 16 เพิ่ม subtitle ในแนวว่า "ทำไมซ่อนปุ่มด้วย `isAdmin && ...` ถึงไม่ใช่การจัดการสิทธิ์"

จุดประสงค์ของการเติม subtitle ไม่ใช่เพื่อให้ชื่อดูยาวขึ้น แต่เพื่อช่วยผู้อ่านจับ "คำถามกลาง" ของบทได้เร็วขึ้นตั้งแต่บรรทัดแรก

### 3. ทำรูปแบบการอ้างอิงในเนื้อหาให้ตรงกัน

ในบท 17 มีการอ้างอิงลักษณะ `ความรู้บทที่ 1, 2, ...` ซึ่งไม่ตรงกับรูปแบบหลักที่ทั้งเล่มใช้ จึงปรับเป็น `ความรู้บท 1, 2, ...` เพื่อให้การอ้างอิงดูเป็นมาตรฐานเดียวกัน

## จุดที่แก้เพราะข้อมูลเชิงเทคนิคคลาดเคลื่อน

ส่วนนี้เป็นงานที่สำคัญ เพราะหนังสือประเภทนี้มีหลายประโยคที่ผูกกับเวอร์ชันของ browser, framework หรือสถานะของมาตรฐาน ถ้าข้อมูลคลาดเคลื่อนแม้เพียงเล็กน้อย คนอ่านสาย senior จะจับได้ทันที

### บท 1

ข้อความเดิมอธิบาย `scheduler.yield` แบบกว้าง ๆ ว่าเริ่มรองรับในช่วงปลายปี 2025 ซึ่งคลุมเครือเกินไป จึงแก้ให้ระบุชัดขึ้นว่าใช้งานได้ใน Chrome 129 ขึ้นไป และ Firefox 142 ขึ้นไป พร้อมเติมคำเตือนว่าควรตรวจ [caniuse](https://caniuse.com/) ก่อนใช้ใน production เพราะ Safari ยังไม่รองรับ

### บท 3

เดิมมีข้อความในทำนองว่า TypeScript 7 เป็นการ "เขียน compiler ใหม่ด้วย Go" ซึ่งสื่อความหมายแรงเกินข้อเท็จจริง จึงปรับเป็นการอธิบายว่า TypeScript กำลังถูก port ไปเป็น native implementation ที่ใช้ Go โดยยังคง semantics เดิมไว้ เหตุผลที่ต้องแก้คือภาษาของทีม Microsoft เองใช้คำว่า port มากกว่า rewrite

### บท 10

ข้อความเดิมบอกว่า RxJS v8 ยังพัฒนาต่ออยู่บน `master` แต่สถานะล่าสุดละเอียดกว่านั้น จึงปรับเป็นว่า v8 ยังอยู่ในช่วง alpha และทิศทางของงานยังรอให้มาตรฐาน TC39 Observable ชัดเจนกว่านี้ก่อน

### บท 12

ประโยคเดิมเขียนว่า Angular resource จะ stable ในช่วง v21-v22 ซึ่งยังกว้างไป จึงแก้ให้เฉพาะเจาะจงขึ้นว่า `httpResource` เข้าแบบ experimental ตั้งแต่ v19.2 และขึ้น stable ใน v22 พร้อมหมายเหตุให้เช็ก changelog ก่อนอ้างในอนาคต

### บท 14

มีสองจุดสำคัญที่ต้องแก้:

1. ข้อความเดิมที่บอกว่า WCAG 2.2 เป็น `ISO/IEC 40500:2025` ไม่ถูกต้อง จึงตัดประโยคนั้นออก และคงไว้เพียงข้อเท็จจริงว่า WCAG 2.2 เป็น W3C Recommendation เวอร์ชันเสถียรล่าสุดตั้งแต่เดือนตุลาคม 2023
2. เคส Carrefour เดิมระบุ timeline ของคดีคลาดเคลื่อน จึงปรับใหม่ให้บอกเฉพาะสาระที่ตรวจยืนยันได้ คือ tribunal de Caen มีคำสั่งให้ทำระบบให้เข้าถึงได้ภายใน 6 เดือน และสถานะคดียังมีการอุทธรณ์อยู่ จึงควรตรวจสถานะก่อนอ้างทุกครั้ง

### บท 16

ข้อความเดิมอธิบาย Trusted Types เหมือนว่าเข้าสู่ Baseline แล้วและ browser หลักรองรับครบ แต่ภาพจริงยังไม่ใช่แบบนั้น จึงแก้เป็นว่า Trusted Types ใช้ได้สมบูรณ์ในฝั่ง Chromium เป็นหลัก ขณะที่ engine อื่นยังตามไม่ครบ และยังไม่ควรอธิบายว่าเป็น Baseline โดยไม่เช็กสถานะล่าสุด

## จุดที่ตรวจแล้วว่าถูก จึงคงไว้

เพื่อไม่ให้บันทึกนี้ดูเหมือนมีแต่การแก้ข้อผิดพลาด จึงบันทึกไว้ด้วยว่ามีหลายประเด็นที่เสี่ยงต่อการล้าสมัย แต่เมื่อตรวจแล้วพบว่ายังคงถูกต้อง จึงไม่ได้แก้ไข ได้แก่:

- เกณฑ์ Core Web Vitals คือ LCP ไม่เกิน 2.5 วินาที, INP ไม่เกิน 200 มิลลิวินาที, และ CLS ไม่เกิน 0.1 ที่ระดับ p75
- INP เข้ามาแทน FID ตั้งแต่ 12 มีนาคม 2024
- แนวโน้มที่ Playwright ได้รับความนิยมสูงขึ้นจนแซง Cypress
- Chrome ยกเลิกแผน phase-out third-party cookie แบบเดิม
- European Accessibility Act มีผลบังคับใช้วันที่ 28 มิถุนายน 2025
- React Compiler 1.0 stable แล้ว
- Angular v22 ออกในเดือนมิถุนายน 2026
- zod v4 และ DTCG spec สถานะที่อ้างในต้นฉบับยังสอดคล้องกับข้อมูลล่าสุด

## งานพิสูจน์อักษรและการเก็บรายละเอียดในเนื้อหา

### บท 2

ตารางเรื่อง coercion มีปัญหาที่แถวสุดท้ายเกี่ยวกับ `NaN == NaN` เพราะมีย่อหน้าอธิบายถูกวางติดอยู่ใน cell เดียวกัน ทำให้ markdown render ตารางเพี้ยน จึงแยกคำอธิบายออกมาไว้เป็นย่อหน้าถัดจากตาราง

### บท 4

มีประโยคหนึ่งในส่วน INP ที่ความคิดสองช่วงชนกันจนอ่านสะดุด จึงเติมเครื่องหมายคั่นเพื่อทำให้จังหวะประโยคชัดขึ้น

### บท 8

แก้ typo จาก `ใส้ต่างะ` เป็น `ไส้ต่าง`

## งานแก้การอ้างอิงข้ามบทและข้ามเล่ม

จุดที่สำคัญที่สุดอยู่ในบท 18 ซึ่งเป็นบทปิดเล่ม

เดิมมีการอ้างถึงหัวข้ออย่าง idempotency, expand-contract, BFF, JVM heap และ story bank โดยเขียนเพียง `(บท 11)`, `(บท 14)`, `(บท 6)`, `(บท 22)`, `(บท 19)` ลอย ๆ ปัญหาคือในเล่ม frontend มีเพียง 18 บท ทำให้บางเลขบทเหล่านี้ชี้ไปคนละความหมาย หรือบางเลขไม่มีอยู่จริงในเล่มนี้

จึงแก้การอ้างอิงทั้งหมดให้ระบุชัดว่าเป็น:

- `เล่ม backend บท 11`
- `เล่ม backend บท 14`
- `เล่ม backend บท 6`
- `เล่ม backend บท 22`
- `เล่ม backend บท 19`

การแก้นี้ช่วยลดความสับสนมาก เพราะคนอ่านจะไม่หลงว่ากำลังถูกชี้กลับไปยังบทในเล่ม frontend เอง

## ประเด็นที่ปิดแล้ว

### บท 3: การจัดลำดับเลขหัวข้อใหม่

เดิมมีหัวข้อย่อยชื่อ `3.8.5 tsconfig...` แทรกอยู่ระหว่าง 3.8 และ 3.9 ซึ่งทำให้โครงสร้างเลขหัวข้อไม่เป็นแพตเทิร์นเดียวกับส่วนอื่นของบท

หลังจากยืนยันแนวทางกับทีมแล้ว จึงแก้เป็น:

- `3.9 tsconfig`
- `3.10 enum vs union`
- `3.11 TypeScript ไม่ได้กัน runtime error`

พร้อมกันนั้นได้อัปเดตจุดอ้างอิงภายในบทที่ชี้ไปยัง section เรื่อง validation จากเดิมที่อ้าง `3.10` ให้เป็น `3.11` ทั้งหมด 3 จุด และตรวจทานซ้ำแล้วว่าลำดับหัวข้อในบท 3 ต่อเนื่องตั้งแต่ 3.1 ถึง 3.11 โดยไม่มีเลขค้าง

## ข้อสรุปเชิงบรรณาธิการ

แม้ผู้เขียนจะบอกว่างานรอบนี้มีส่วนของ "เติมตัวอย่างจุดบาง" อยู่ด้วย แต่เมื่ออ่านครบทั้งเล่มแล้ว พบว่าเนื้อหาส่วนใหญ่มีตัวอย่างโค้ดรองรับประเด็นสำคัญอยู่แล้ว โดยเฉพาะจุดที่อธิบายกลไกยาวหรือมี trade-off เยอะ

ดังนั้น งานบรรณาธิการรอบนี้จึงไม่ได้มุ่งเพิ่มตัวอย่างใหม่จำนวนมาก แต่เน้นทำให้ของเดิม:

- อ่านต่อเนื่องขึ้น
- อ้างอิงกันได้ชัดขึ้น
- ใช้ภาษาที่สม่ำเสมอขึ้น
- แม่นยำพอสำหรับผู้อ่านระดับ senior
