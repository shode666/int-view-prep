# บทที่ 17 — Leadership & Behavioral Interview

> ตำแหน่ง Dev Lead ตกรอบกันตรงนี้มากกว่ารอบ technical — เพราะรอบนี้เขาไม่ได้วัดว่าคุณเก่งไหม แต่วัดว่า **ให้ทีมอยู่กับคุณแล้วทีมดีขึ้นหรือแย่ลง**

## เข็มทิศก่อนอ่าน

หลังจากคุยเรื่องระบบมาหนักหลายบท บทนี้เปลี่ยนสนามจาก code ไปสู่คน แต่ใช้ตรรกะเดียวกันคือดูผลลัพธ์จากพฤติกรรมจริง ไม่ใช่จากคำอธิบายสวย ๆ Behavioral interview ที่ดีจึงไม่ได้ถามว่า "คุณเป็น lead แบบไหน" แต่ถามผ่านเหตุการณ์ว่าเมื่อมี conflict, pressure หรือความผิดพลาด คุณทำให้ทีมดีขึ้นจริงหรือไม่

ให้อ่านบทนี้แบบเตรียมของจริงสำหรับการเล่าเรื่อง ไม่ใช่แบบอ่านทฤษฎี เพราะคุณค่าของบทนี้อยู่ที่การแปลงประสบการณ์กระจัดกระจายให้กลายเป็นคำตอบที่ชัดและน่าเชื่อ

## หลักการตอบ: STAR + แผลจริง

**S**ituation (บริบทสั้นๆ) → **T**ask (หน้าที่/เดิมพันของคุณ) → **A**ction (คุณทำอะไร — ขยายส่วนนี้ 60%) → **R**esult (ผลวัดได้ + สิ่งที่เรียนรู้)

กติกาที่ทำให้คำตอบดูจริง:

- ใช้ "ผม" ไม่ใช่ "เรา" ในส่วน Action — interviewer อยากรู้ว่า **คุณ** ทำอะไร ไม่ใช่ทีมทำอะไร
- มีตัวเลขอย่างน้อยหนึ่งตัวใน Result (ลด deploy time จาก 2 ชม. เหลือ 15 นาที, Mean Time To Recovery (MTTR) ลงครึ่ง, คนที่เกือบลาออกอยู่ต่ออีก 2 ปี)
- เรื่องที่ดีที่สุดมักเป็นเรื่องที่ **มีของเสียอยู่ในนั้น** — ตัดสินใจพลาดแล้วแก้ยังไง น่าเชื่อกว่าเรื่องที่ perfect ทุกจังหวะ
- จบทุกเรื่องด้วย "สิ่งที่ผมทำต่างไปหลังจากนั้น" — นี่คือหลักฐานว่า senior จริง (เรียนรู้จากแผล ไม่ใช่แค่รอดมาได้)

## หมวดคำถามที่ออกแน่ + สิ่งที่เขาวัดจริงๆ

### 1. Conflict ในทีม

*"เล่าตอนที่ขัดแย้งกับเพื่อนร่วมทีม/สถาปนิก/ทีมอื่นเรื่อง technical"*
เขาวัด: แยก "แพ้-ชนะ" ออกจาก "ถูก-ผิด" ได้ไหม — คำตอบดี: ฟังก่อน, หา objective criteria (benchmark, prototype, data — ไม่ใช่เสียงดัง), ยอมแพ้เป็นเมื่อ evidence ชี้อีกทาง, และ **disagree and commit** เมื่อตัดสินแล้ว
กับดัก: เล่าแล้วอีกฝ่ายเป็นตัวร้ายล้วน — ไม่มีเรื่อง conflict จริงที่อีกฝ่ายผิด 100%

### 2. คน underperform

*"ทีมมีคนทำงานไม่ถึงมาตรฐาน จัดการยังไง"*
เขาวัด: ความเป็นธรรม + ความกล้า — โครงตอบ: (1) หาสาเหตุก่อนตัดสิน — คุย 1:1 ว่าติดอะไร (ปัญหาส่วนตัว? งานไม่ตรง skill? expectation ไม่ชัด?) (2) ตั้ง expectation ชัด + ระยะเวลา + support ที่ให้ (pairing, ลด scope, mentor) (3) ติดตามเป็นลายลักษณ์ (4) ถ้าไม่ดีขึ้น — escalate ตามกระบวนการ ไม่ปล่อยให้ทีมแบกเงียบๆ
กับดัก: ตอบข้ามไปข้อ 4 ทันที (โหด ไม่ fair) หรือไม่มีข้อ 4 เลย (ใจดีจนทีมพัง) — และห้ามลืมว่า **ปล่อยคน underperform ไว้คือการลงโทษคนที่ perform**

### 3. เถียงกับ Product / ผู้บริหาร

*"PM จะเอา feature พรุ่งนี้ แต่ code ยังไม่พร้อม"* / *"เคยผลักดัน tech debt ยังไง"*
เขาวัด: แปลภาษา technical เป็นภาษาธุรกิจได้ไหม — tech debt ห้ามขายว่า "code สกปรก" ต้องขายเป็น **ความเสี่ยงและต้นทุน**: "feature ในโมดูลนี้ใช้เวลาเพิ่ม 40% ทุก sprint และ bug rate สูงสุดในระบบ — ขอ 2 sprint แลกกับ velocity ที่กลับมา"
ท่ามาตรฐาน: เสนอ option พร้อม trade-off ให้เขาเลือก (ship พรุ่งนี้แบบ feature flag + จำกัด user 5% / เลื่อน 3 วันได้ของเต็ม) — lead ที่ดีไม่ตอบ "ไม่ได้" แต่ตอบ "ได้ ถ้าแลกกับ X"
กับดัก: เล่าแบบ dev ชนะ PM แพ้ — เขากำลังจะจ้างคุณไปทำงาน **ร่วมกับ** PM

### 4. Mentoring / สร้างคน

*"พัฒนา junior ยังไง"* / *"delegate ยังไง"*
เขาวัด: คุณ scale ตัวเองผ่านคนอื่นได้ไหม (นี่คือความต่างระหว่าง senior dev กับ lead) — วัตถุดิบตอบ: code review เป็นเครื่องมือสอนไม่ใช่ด่าน, มอบงานที่ **ยากกว่าที่เขาคิดว่าตัวเองทำได้นิดหนึ่ง** พร้อม safety net, ถามนำแทนบอกคำตอบ, ให้ credit ต่อหน้าคน ให้ feedback ลับหลังคน
Delegation: มอบ **outcome ไม่ใช่ขั้นตอน** + ระดับความเสี่ยงกำหนดระดับการตาม (ของ critical ตามถี่ ของ safe ปล่อย) — และงานที่คุณทำเองเร็วกว่า คืองานที่ควร delegate ที่สุด (ไม่งั้นทีมไม่โตและคุณเป็นคอขวดตลอดกาล)

### 5. Failure / Incident

*"เล่า production incident ที่หนักสุด"* / *"การตัดสินใจที่ผิดพลาดที่สุด"*
เขาวัด: ownership — คำตอบดีมีสามชั้น: ตอนไฟไหม้ทำอะไร (mitigate ก่อน root cause — เอาระบบกลับมาก่อนค่อยสืบ), หลังไฟดับทำอะไร (blameless postmortem — โฟกัส process ไม่ใช่หาคนผิด), ระบบเปลี่ยนอะไรถาวร (alert ใหม่, runbook, กฎ deploy)
กับดัก: เรื่องที่ความผิดเป็นของคนอื่น/ระบบเก่า/ทีม infra ทั้งหมด — ต้องมีส่วนที่ **คุณ** พลาดอยู่ในเรื่อง ไม่งั้นถือว่าหลบคำถาม

### 6. Prioritization / ทำงานภายใต้ข้อจำกัด

*"งานล้น ทีมเล็ก เลือกยังไง"*
เขาวัด: กล้าตัดไหม — คำตอบดีมีคำว่า "ไม่ทำ" อยู่ด้วยเสมอ: จัดตาม impact × effort, สื่อสารสิ่งที่**ไม่ได้เลือก**ให้ stakeholder รู้พร้อมเหตุผล (การ deprioritize เงียบๆ คือระเบิดเวลา), กัน capacity ให้ on-call/bug/debt ก่อนรับ feature เต็มมือ

## คำถามเฉพาะตำแหน่ง Lead ที่คนไม่ค่อยเตรียม

- *"Lead กับ Senior ต่างกันยังไงในมุมคุณ"* — คำตอบที่ดีสักแบบ: senior = ตัวคูณผ่าน code และ design, lead = ตัวคูณผ่าน **คนและการตัดสินใจ** — ความสำเร็จของ lead วัดจากสิ่งที่ทีมทำได้โดยไม่ต้องมีคุณอยู่ในห้อง
- *"จะทำอะไรใน 90 วันแรก"* — เดือนแรก: ฟัง+วาดแผนที่ (ระบบ, คน, pain) ห้ามรื้ออะไร · เดือนสอง: quick win ที่ทีมเจ็บสุด (มัก CI/CD หรือ alert noise) · เดือนสาม: เสนอแผนระยะยาวด้วย data ที่เก็บมา
- *"ถ้าทีมไม่เห็นด้วยกับ direction ที่คุณเลือก"* — แยกสองเคส: ยังไม่ตัดสิน (เปิด Request for Comments (RFC) / Architecture Decision Record (ADR) ให้เถียงด้วย evidence) vs ตัดสินแล้ว (อธิบายเหตุผล, รับ feedback loop, แต่เดินหน้า — ทีมที่ revisit ทุกการตัดสินใจคือทีมที่ไม่ไปไหน)
- *"เคย fire คนไหม / เคยอยู่ในกระบวนการนั้นไหม"* — ตอบตรงถ้าเคย เน้นความ fair ของกระบวนการ + สิ่งที่ทำให้ไม่ต้องไปถึงจุดนั้นอีก (hiring ดีขึ้น, expectation ชัดตั้งแต่แรก)

## Red flags ที่ทำให้ตกทันที

พูดถึงทีมเก่า/เจ้านายเก่าแบบขยะ · ทุกเรื่องตัวเองเป็นฮีโร่ · ไม่มีเรื่องไหนที่ตัวเองผิด · "ผมทำเองหมดเพราะทีมช้า" (ตรงข้าม lead) · ตอบทุกข้อเป็นทฤษฎีไม่มีเหตุการณ์จริง · ไม่เคยถามอะไรกลับเลย

## เตรียมยังไง

เขียน **story bank 6–8 เรื่อง** จากประสบการณ์จริง (template อยู่บท 19) — หนึ่งเรื่องดีๆ ใช้ตอบได้ 3–4 คำถาม (incident เรื่องเดียว ตอบได้ทั้ง failure, conflict, prioritization, สอนทีม) แล้วซ้อมเล่าออกเสียงให้จบใน 2–3 นาทีต่อเรื่อง

บทต่อไป: อีกครึ่งของรอบ onsite — System Design Interview ให้เป็นระบบใน 45 นาที

## สรุปท้ายบท

- behavioral interview วัดผลกระทบของคุณต่อคนและทีม ไม่ใช่แค่ความสามารถเชิงเทคนิคส่วนตัว
- คำตอบที่ดีต้องมีเหตุการณ์จริง การกระทำที่เป็นของคุณ และผลลัพธ์ที่ชัดพอจะเชื่อได้
- ความเป็น senior โผล่ชัดที่สุดในจุดที่คุณยอมรับความพลาด เรียนรู้จากมัน และเปลี่ยนพฤติกรรมหลังจากนั้น
- บทนี้จึงไม่ได้สอนให้พูดสวย แต่สอนให้เล่าเรื่องจริงอย่างมีโครงและมีน้ำหนัก

## ก่อนไปบทถัดไป

เมื่อมีเรื่องเล่าสำหรับฝั่งคนแล้ว บทถัดไปจะกลับไปสนาม technical interview อีกครั้ง โดยโฟกัสที่การเดิน system design ให้เป็นขั้นและมี trade-off ชัดในเวลาจำกัด
