# My Answers — ฉบับตกผลึกของผมเอง

> คำตอบที่ผมสร้างเองระหว่างซ้อม (2026-07-02) — เปิดทวน 10 นาทีก่อนเข้าห้องสัมภาษณ์
> ทุกอันผ่านการเล่าออกเสียงแล้ว ห้ามท่องใหม่ ให้เล่าจากความเข้าใจ

## SOLID ห้าตัว (เวอร์ชันภาษาอังกฤษของผม)

- **S** — A class should have only one reason to change. *(วัดจาก "ใครมาขอแก้" — สองทีมแก้ไฟล์เดียวกัน = ผ่า)*
- **O** — Open for extension, closed for modification. *(ของใหม่ = เพิ่ม code ใหม่เข้าจุดเสียบ ไม่ผ่าของเก่า — แต่ทำเมื่อเห็นการแก้ซ้ำครั้งที่ 3 ไม่ใช่เผื่อ)*
- **L** — Subtypes must be substitutable for their parent: **no missing behavior, no lies, no surprises** (ผ่านสายตาแม่ — มี method เกินได้ แต่ห้ามเปลี่ยนพฤติกรรมที่แม่สัญญา) *(ตรวจด้วย contract test — type checker จับไม่ได้)*
- **I** — Keep interfaces small, **shaped by what the consumer needs** (ไม่ใช่ตามที่ provider มี)
- **D** — Services never call infra directly; they communicate through a **business-owned interface** — business ประกาศว่าต้องการอะไร ฝั่ง infra ไปหาทางทำมา business ไม่ยุ่งว่าได้มาจากไหน → test ได้โดยไม่มี DB, เปลี่ยน DB โดยไม่แตะ business

## โมเดลกล่อง-ข้อต่อ (ของผมเอง — วาดบน whiteboard ได้)

```
┌─────────┐          ┌─────────┐
│ กล่อง=S │ ──────▶  │ กล่อง=S │      ข้อต่อ = I, D
└─────────┘  ข้อต่อ   └─────────┘      ของที่เสียบต้องเชื่อถือได้ = L
              I,D                      O = กฎคร่อมทั้งคู่: กล่องปิด ข้อต่อเปิด
```

- **"ลัดวงจร" ที่ O ห้าม**: (1) บัดกรีเข้ากล่องเก่า (เพิ่ม elif ใน function เดิม) (2) เดินสายอ้อมข้อต่อ (`new PostgresRepo()` กลางโค้ด)
- **SOLID คุมกันเองเป็นวงจร**: O เปิดรูเสียบให้โต → เสียบไปเรื่อยๆ จนบวม → S/I กลับมาตรวจว่า "ของที่แปะเข้ามา ต้องแยกกล่อง/แยกสัญญาแล้วไหม" — ไม่มีตัวไหนยืนคนเดียวได้
- **Map กับบท 2**: ข้างในกล่อง = cohesion (S คุม) / ข้อต่อ = coupling (L,I,D คุม) — "ข้างในแน่น ข้างนอกหลวม"

## เรื่องร้านอาหาร (ใช้เล่า SOLID ทั้งชุดใน 1 นาที)

ครัวแบ่งสถานี สถานีละหน้าที่ (S) · เมนูใหม่ = เพิ่มการ์ดสูตร ไม่รื้อครัว (O) · เชฟแทนกะต้องทำตามสูตรครบเหมือนเดิม (L) · แต่ละสถานีได้ใบสั่งเฉพาะเมนูตัวเอง ไม่ใช่เล่มทั้งร้าน (I) · ร้านเขียนสเปควัตถุดิบ แล้ว supplier ปรับตัวมาตามสเปค — ไม่ใช่ร้านเปลี่ยนเมนูตามของที่ supplier มี (D)

> สรุปประโยคเดียว: SOLID คือการจัดการ "สัญญา" ระหว่างส่วนต่างๆ ให้การเปลี่ยนแปลงราคาถูกที่สุด — S/I ทำสัญญาให้เล็กและเรื่องเดียว, L บังคับรักษาสัญญา, D กำหนดเจ้าของสัญญา, O คือรางวัล

## D / DIP — แก่นที่ต้องพูดได้

- ลูกศร dependency = บรรทัด `import` — แก้ของที่ถูกชี้ คนชี้พังตาม → ของผันผวนต้องเป็นฝ่ายชี้ไปหาของเสถียร
- **Inversion = ใครถือปากกา**: interface อยู่บ้านผู้ใช้ (business) — supplier เขียน spec เอง = decoupling เฉยๆ / ร้านเขียน spec = DIP
- Spring DI คือ "คนต่อสาย" ไม่ใช่ตัว DIP — ทิศของ import ต่างหากที่เป็น
- ศัพท์อังกฤษ: **port** (interface ที่ business ประกาศ) / **adapter** (ตัว implement ฝั่ง infra) / **boundary** (รอยต่อเชิงสถาปัตยกรรม) / **seam** (จุดฉีด fake — บริบท testing/legacy)

## Layer 4 วลี (ตอบ "อธิบาย layered architecture")

> **Presentation แปลง · Application ประสาน (แต่ไม่ตัดสิน) · Domain ตัดสิน (แต่ไม่แตะโลก) · Infra ทำตามใบสั่ง**

- ทุกชั้น import domain ได้ / domain ไม่ import ใครเลย (ลบ infra ทิ้ง domain ต้อง compile ผ่าน)
- Transaction boundary = application (use case) และห้ามมี external call ใน transaction
- ฝั่งเขียนบังคับผ่าน use case ทุกทางเข้า (HTTP/consumer/batch) / ฝั่งอ่านล้วนลัดได้ (CQRS เบา)
- CRUD ล้วน = ยุบเหลือ 2 ชั้น (handler+repo) — ชั้นกลางว่างเปล่าไม่ต้องสร้าง คลี่กลับเมื่อ business rule มาจริง
- Repository ที่ทำถูก = SOLID ครบห้าตัวในของชิ้นเดียว → "ถ้าผมอธิบาย repository ได้ ผมอธิบาย SOLID ได้ทั้งชุดด้วยตัวอย่างเดียว"

## Story ของผม: EAR ยุค Struts → Spring

ระบบที่ผมทำ: WAR ดึง interface จาก JAR (db) ผ่าน container — decoupling สำเร็จแต่ inversion ไม่ครบ เพราะ interface อยู่บ้านฝั่ง DB (classloader ของ EAR บังคับ: JAR มองไม่เห็น class ใน WAR) ทางแก้ยุคนั้นคือ api.jar ตรงกลาง — ถ้าทำวันนี้ผมจะให้ business ถือสัญญาตั้งแต่แรก แบบที่ Spring ยุคนี้ทำได้ด้วย package แยก
*(ใช้ตอบ: เล่า pattern จากงานจริง / สิ่งที่จะทำต่างไป / พิสูจน์ว่าเข้าใจไม่ใช่ท่อง)*

## กฎการตอบทุกข้อ

**ปัญหาที่มันแก้ → ถ้าไม่มีจะเกิดอะไร → ราคาที่จ่าย → จุดพังที่เคยเห็น** — และพูด trade-off เองโดยไม่ต้องถูกถาม
