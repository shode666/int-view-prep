# บทที่ 18 — System Design Interview Framework

> เนื้อหาเชิงลึกอยู่บท 2–11 และ 15 แล้ว — บทนี้คือ **วิธีเอาของพวกนั้นออกมาขายใน 45 นาที** บน whiteboard/Excalidraw ซึ่งเป็นเกมคนละแบบกับการออกแบบจริง: เขาไม่ได้อยากได้ระบบสมบูรณ์ เขาอยากเห็น **วิธีคิดเป็นขั้น + trade-off ทุกทางเลือก**

## เข็มทิศก่อนอ่าน

บทนี้เป็นสะพานระหว่างความรู้เชิงเทคนิคกับความสามารถในการสื่อสารออกมาให้ interviewer เห็น หลายคนออกแบบระบบได้ แต่ตกรอบเพราะเล่าไม่เป็นขั้น ไม่เปิด trade-off เอง หรือใช้เวลาไปกับรายละเอียดที่ไม่เปลี่ยนคำตอบระดับระบบ

ให้อ่านบทนี้เหมือนกำลังซ้อมเดินโจทย์ต่อหน้าคนอื่น ทุก step มีไว้เพื่อควบคุมจังหวะการคิดและทำให้เหตุผลของเรามองเห็นได้ ไม่ใช่เพื่อท่องเป็นพิธีกรรม

## โครงเวลา 45 นาที

| นาที | ทำอะไร | ห้ามทำอะไร |
|---|---|---|
| 0–7 | Clarify requirements + ตีกรอบ scope | ห้ามวาดอะไรก่อนถาม |
| 7–12 | Capacity estimation (คร่าวๆ พอชี้ทาง) | ห้ามจมกับเลขเป๊ะ |
| 12–17 | API หลัก + data model | ห้ามลงทุก endpoint |
| 17–32 | High-level design แล้วเจาะลึก 1–2 จุดที่ interviewer สนใจ | ห้าม monologue — เช็คทิศทุก 3–4 นาที |
| 32–42 | Scale + failure mode + trade-off | ห้ามเสนอของ hype โดยไม่มีเหตุผล |
| 42–45 | สรุป + จุดอ่อนที่รู้ตัว + ถ้ามีเวลาจะทำอะไรต่อ | ห้ามจบแบบ "ก็ประมาณนี้ครับ" |

## ขั้น 1: Clarify (นาที 0–7) — ขั้นที่คนตกมากที่สุด

โดดไปวาดเลย = สัญญาณ junior ชัดที่สุด คำถามมาตรฐานที่ต้องถาม:

- **Functional**: ใครใช้ ทำอะไรได้บ้าง — แล้ว **ตัด scope เอง**: "ผมโฟกัส core flow สามอันนี้ก่อน ที่เหลือถ้ามีเวลา" (การกล้าตัดคือคะแนน ไม่ใช่การเสียของ)
- **Scale**: user เท่าไหร่ Daily Active Users (DAU) เท่าไหร่ read:write ratio ประมาณไหน — เลขพวกนี้เปลี่ยน design ทั้งกระดาน (1K กับ 10M user คือคนละระบบ)
- **NFR (Non-Functional Requirements) ที่ต้องถามเสมอ**: consistency ยอม eventual ได้ตรงไหน (บท 11), latency เป้าเท่าไหร่, availability สำคัญแค่ไหน — และคำถามเด็ด: **"ระบบนี้พังหนึ่งชั่วโมง ธุรกิจเสียอะไร"** — คำตอบกำหนดงบความซับซ้อนทั้งหมด

## ขั้น 2: Capacity Estimation (นาที 7–12)

จุดประสงค์ไม่ใช่เลขถูก — คือโชว์ว่า **ตัดสินใจด้วยตัวเลข ไม่ใช่ความรู้สึก** ตัวเลขที่ต้องพกเข้าห้อง:

- 1 วัน ≈ 86,400 วินาที (ปัดเป็น 100K ให้หารง่าย) → **10M requests/วัน ≈ 115 Requests Per Second (RPS) เฉลี่ย → peak คูณ 3–5 ≈ 500 RPS** — เลขระดับนี้ Postgres เครื่องเดียว + replica เอาอยู่สบาย (ข้อสรุปนี้แหละคือคะแนน: ไม่ลาก Kafka/sharding มาใส่โจทย์ที่ไม่ต้องการ)
- Latency ที่ควรรู้ขนาดสัมพัทธ์: memory ~100ns · SSD ~100µs · Round-Trip Time (RTT) ใน datacenter ~0.5ms · query มี index ~1–10ms · ข้าม region ~50–150ms — ไว้ตอบว่าทำไมต้องมี cache หรือทำไมไม่ต้องมี
- Storage: ประเมิน row size × ปริมาณ × retention — "order ละ ~1KB, วันละ 1M = 1GB/วัน = 365GB/ปี → ยังไม่ต้อง shard แต่ควรวางแผน archive"

## ขั้น 3: API + Data Model (นาที 12–17)

- เขียน endpoint หลัก 3–5 เส้นพอ (REST ตามบท 6 — ระบุ idempotency ของ POST ที่เงินเกี่ยวด้วย จะได้คะแนนบท 11 ฟรี)
- Data model: ตาราง/collection หลัก + คีย์ + ความสัมพันธ์ — แล้วชี้ **จุดที่จะมีปัญหา**: "ตาราง order โตเร็วสุด index ตาม query pattern นี้, hot row ตรง stock ต้องใช้ conditional update (บท 7)"
- เลือก DB พร้อมเหตุผลเป็นประโยคเดียว: "เริ่ม Postgres เพราะ transaction จริงและทีมรู้จักดี — จะพิจารณา NoSQL เมื่อ access pattern เป็น key-value ล้วนและ scale เกิน X" — ห้ามตอบ "MongoDB เพราะ flexible"

## ขั้น 4: High-level + เจาะลึก (นาที 17–32)

วาด: client → gateway (auth, rate limit — บท 6, 10) → services ตาม business capability (บท 15) → DB per service + queue ตรง async boundary (บท 9)

จากนั้น**ถาม interviewer ว่าอยากเจาะไหน** — สัญญาณ collaboration ที่เขาชอบมาก จุดเจาะยอดนิยมและบทที่หนุน:

- เขียนพร้อมกันเยอะ / ตัด stock → lock strategy (บท 7)
- ส่ง event ยังไงให้ไม่หาย → outbox + at-least-once + idempotent consumer (บท 9, 11)
- flow หลายขั้นพังกลางทาง → saga + compensation (บท 11)
- อ่านหนัก → cache ชั้นไหน + invalidation + stampede (บท 6)

ระหว่างเจาะ พูด trade-off เสมอ ("ทางเลือก A ง่ายกว่าแต่ตายที่ scale นี้, ผมเลือก B เพราะ requirement ข้อนี้") — **ประโยครูปนี้แหละคือสิ่งที่เขาจดลง scorecard**

## ขั้น 5: Scale + Failure (นาที 32–42)

ไล่ตามลำดับต้นทุน (อย่าเปิดด้วย shard!): vertical scale → read replica + cache → horizontal scale ตัว service (stateless — session ไป Redis) → partition/shard เป็นทางเลือกสุดท้ายพร้อมบอกราคา (rebalance, cross-shard query, hot partition)

แล้วยิง failure mode เองก่อนถูกถาม (บท 10 ทั้งบท): "จุดตายคือ X, กันด้วย timeout/breaker/bulkhead, DB ตัวเดียว = Single Point of Failure (SPOF) ใส่ replica + failover, และผมขอ monitoring สามตัวนี้ก่อนเลย: RED metrics (Rate, Errors, Duration — บท 20), DLQ depth, saga stuck age (บท 15)"

## ขั้น 6: ปิด (นาที 42–45)

สรุป 30 วินาที + **จุดอ่อนที่รู้ตัว**: "design นี้แลก consistency มาเป็น availability ตรงจุดนี้, ยังไม่ได้คิดเรื่อง multi-region, ถ้ามีเวลาต่อจะทำ capacity ละเอียดตรง Y" — การประกาศจุดอ่อนเองคือความมั่นใจ ไม่ใช่ความอ่อน

## สัญญาณที่ทำให้ผ่าน / ตก

**ผ่าน**: ถามก่อนวาด · ตัดสินใจด้วยตัวเลข · ทุกทางเลือกมี "เพราะ...แลกกับ..." · เริ่มง่ายแล้ว scale เมื่อโจทย์บังคับ · คุยเป็น dialogue · ยกแผลจริงประกอบ ("เคยเจอ pool exhaustion แบบนี้ วิธีที่ได้ผลคือ...")

**ตก**: สาด technology ทันที (Kafka + K8s + microservice 12 ตัว สำหรับ 100 user) · เงียบคิดนานๆ ไม่เล่าว่าคิดอะไร · ยึด design แรกจนจบแม้ interviewer ใบ้ว่ามีปัญหา (เขา**ตั้งใจ**ใบ้ — การรับ hint แล้วปรับคือสิ่งที่เขาทดสอบ) · ตอบทุก follow-up ว่า "ก็ใส่ cache" 

## ซ้อมยังไง

เอาโจทย์คลาสสิก (URL shortener, rate limiter, order system — ใช้บท 15 ได้เลย, chat, news feed) จับเวลา 45 นาทีวาดจริงพูดออกเสียงจริง — ความต่างระหว่างรู้กับเล่าได้ในเวลาจำกัด ใหญ่กว่าที่คิดมาก

บทต่อไป: เตรียมกระสุนส่วนตัว — Story Bank, intro pitch และคำถามย้อนถาม

## สรุปท้ายบท

- system design interview ไม่ได้วัดแค่คำตอบสุดท้าย แต่วัดวิธีคิดที่ถูกเผยออกมาระหว่างทาง
- การจัดเวลา, การ clarify scope และการเปิด trade-off เองคือส่วนสำคัญพอ ๆ กับความถูกต้องของ solution
- คนที่ทำได้ดีมักไม่ใช่คนที่จำ architecture template เยอะที่สุด แต่คือคนที่ควบคุมจังหวะการสนทนาและรับ hint ได้ดี
- บทนี้จึงเปลี่ยนความรู้ทางเทคนิคจากบทก่อนหน้าให้กลายเป็นรูปแบบการสื่อสารที่ใช้ในห้องสัมภาษณ์จริง

## ก่อนไปบทถัดไป

เมื่อโครงตอบ system design เริ่มลงตัวแล้ว บทถัดไปจะช่วยเตรียมของส่วนตัวที่ต้องมีติดตัวไว้เสมอในรอบสัมภาษณ์ เช่น story bank, intro pitch และคำถามย้อนถาม
